Target DB architecture for [[printers-companion]] (decided 2026-06-16). Engine/pool/indexes/JSON are already fine; these two were "как попало" and are the conventions to follow.

## Commits — unit of work per boundary
- **One transaction per unit** (one HTTP request, or one background job). Commit ONCE at the boundary; rollback on exception. No partial mid-unit commits.
- `get_db` dependency: `yield db` → `db.commit()` on success → `except: db.rollback(); raise` → `finally: db.close()`.
- `session_scope()` contextmanager in `storage/db/session.py` for **non-request** code (background tasks, workers, startup import, ad-hoc): same commit/rollback/close. Replaces bare `SessionLocal()` + manual `db.commit()`.
- **Repositories NEVER commit.** They add/mutate and `flush()` only when an autogenerated PK is needed within the unit. The old `Repository.commit()` (→ `self.db.commit()`) was the anti-pattern: ~40 scattered `repo.commit()` calls meant partial writes on later failure. Rename repo `commit()`→`flush()`.

## Auto-migration — schema auto-update
- **Single source of truth = Alembic.** `create_all()` is ONLY for the test harness (conftest), never the prod/runtime mechanism.
- Migrations authored with explicit DDL / `alembic revision --autogenerate` (env.py already sets `target_metadata = Base.metadata`). NEVER `Base.metadata.create_all()` inside a migration — that's why 0001/0002 are facades that never ALTER existing tables (see [[m450-print-time-project]] for the jump_speed drift bug fixed by 0003).
- `storage/db/migrate.py::upgrade_to_head()` runs `alembic upgrade head` programmatically (Config from alembic.ini, abs script_location; env.py injects the url from settings).
- **Prod/docker:** run once in the entrypoint BEFORE workers spawn — `command: sh -c "alembic upgrade head && uvicorn … --workers N"`. Race-free (single process). Lifespan must NOT migrate in prod (4 workers would race).
- **Local dev:** lifespan runs `upgrade_to_head()` when `app_env=="local"` (single process → safe). Replaces the old `create_all()`-on-local in lifespan.
- Drift guard (recommended follow-up): a test that runs `alembic upgrade head` on a fresh DB and asserts the schema matches `Base.metadata`.
